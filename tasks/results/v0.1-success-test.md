# Artha v0.1 — success test results

**Date:** 2026-06-21 · **Verdict: PASS** (SPEC "Done when" #8).

An agent given Artha's `context_for_task` applied the proof repo's conventions
using **56% fewer discovery tool-calls** (34 → 15) than the same agent without
Artha — clearing the locked **≥30%** bar — while both arms correctly applied the
conventions and **nothing was pasted into the prompt**.

## What was tested

The locked baseline is in [10-success-test.md](../10-success-test.md). In short:

- **Task (identical both arms):** "Add a new topic page about *Rate Limiting* to
  this engineering-notes app, following all of this codebase's existing
  conventions." No conventions in the prompt.
- **Proof repo:** [brijesh-engineering-notes](file:///C:/Code/brijesh-engineering-notes)
  — a real React/Vite + Cloudflare Workers content-site SPA, 299 commits.
- **A/B:** same agent (`claude -p`, headless), same prompt, identical fresh copies
  of the repo with `CLAUDE.md` + `CONTENT-GUIDE.md` removed so conventions must be
  *discovered*. **Arm A:** no MCP. **Arm B:** the Artha MCP server serving a
  certified index of the repo's conventions.
- **Metric:** discovery tool-calls = `Read` + `Grep` + `Glob` + read-like `Bash`
  (`cat`/`grep`/`ls`/…) — the calls an agent makes to *find* conventions. Calls to
  the Artha tool are counted separately.
- **Pass:** Arm B ≥30% fewer discovery calls **and** conventions applied.

|---|---|---|---|
| **Discovery tool-calls** | **34** | **15** | **−56%** |

| — via read-like `Bash` | 21 | 0 | |
| `context_for_task` calls | 0 | 1 | |
| Total tool calls | 42 | 25 | −40% |
| Conventions applied | ✅ | ✅ | |

Arm B replaced Arm A's scattered exploration (21 `cat`/`grep`/`ls` shell calls +
13 tool reads) with **one** `context_for_task` call (`"Add a new topic page about
Rate Limiting following existing conventions: topic data module, page component,
navigation wiring, section content standard"`) plus a handful of confirmatory
reads of one example file.

Per-arm tool tallies and the agents' own summaries are in
[runs/armA.summary.json](runs/armA.summary.json) and
[runs/armB.summary.json](runs/armB.summary.json).

## Conventions applied (both arms)

Both arms produced a correct, convention-following result — Arm B with far less
discovery:

- `src/data/rateLimiting.ts` — exports `…Intro` / `…Sections: SystemArchSection[]`
  / `…Quiz` (the data-module convention), with ≥1 Indian-company example
  (Razorpay, NPCI, Zerodha, Flipkart, …) per the content standard.
- `src/pages/RateLimitingPage.tsx` — a thin 19-line page component.
- A `<Route>` added to `src/App.tsx` and an entry added to `src/components/Sidebar.tsx`.

## What this shows

The headline v0.1 claim holds: serving certified meaning over MCP let the agent
apply the team's conventions **without them in the prompt** and with **measurably
less rediscovery**. The win is concentrated where Artha is strongest — one ranked,
budgeted tool call replaces a sprawl of greps and file reads.

## Threats to validity (honest caveats)

- **Single run per arm.** `claude -p` is non-deterministic; these are one matched
  pair, not a distribution. The margin (56%) is comfortably above the bar, but
  treat the exact number as indicative.
- **`REVIEW-GUIDE.md` was left in both arms** (the locked protocol removed only
  `CLAUDE.md` + `CONTENT-GUIDE.md`). Both arms could and did read it for the
  content standard. This is a *constant* across arms, so the delta is fair — and
  it makes the result **conservative**: removing it would force Arm A to discover
  more from code, widening the gap.
- **The agent only reliably called Artha once the server advertised usage
  `instructions`** (see below). Without them, an earlier run ignored the tool and
  showed no benefit. Real deployments should keep the instructions (Artha ships
  them) or nudge via the project's agent config.
- **Mining yield is low on a *content* repo:** of 299 commits only 3 carry
  rationale signals, and 10 sampled commits all correctly classified as
  "no decision." `artha mine` runs and classifies real commits correctly; the
  certified value here comes from hand-authored conventions, not mined decisions.

## Fixes this test drove (now in the product)

- **MCP server `instructions`** — Artha tells the agent to call `context_for_task`
  before rediscovering conventions ([src/mcp/server.ts](../../src/mcp/server.ts)).
- **`ARTHA_REPO_ROOT`** — resolve the index when a client doesn't launch the
  server from the repo root.
- Harness/README: MCP config paths must be **OS-native** (Windows paths inside
  `.mcp.json`), and the certified index must be served from a **sidecar** dir, not
  left readable as files in the repo.

## Reproduce

```bash
# builds dist/, mines/certifies are pre-authored in tasks/results/proof-repo-fixture/.artha
node dist/cli.js --version
ARTHA_DIR=/c/Code/artha PROOF=/c/Code/brijesh-engineering-notes \
  bash scripts/success-test.sh "Rate Limiting"
```

The harness ([scripts/success-test.sh](../../scripts/success-test.sh)) builds the
sidecar index from the certified fixture, runs both arms, and prints the counts
via [scripts/count-tools.mjs](../../scripts/count-tools.mjs).
